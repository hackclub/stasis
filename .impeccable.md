## Design Context

### Users

Stasis is a hardware hackathon platform for Hack Club — teens 13–18 from around the world, culminating in an in-person 4-day build event in Austin, TX. The audience is **dual-cohort and both must be served at once**:

- **Beginners land.** A 13-year-old who has never soldered should feel welcomed, not talked down to. They need clear paths, low-risk entry points (starter projects), and an interface that signals "you belong here."
- **Experienced makers stay.** A teen who already CADs and reflows boards should feel respected — no cartoon mascots, no kiddie gamification, no patronizing tone. Depth and craft should reward them.

Context of use: extended sessions on the laptop (picking projects, logging hours, reviewing journals), with mobile check-ins and NFC/inventory usage during the in-person event.

### Brand Personality

**Retrofuturistic single-color console.** Three words: **serialized, instrumented, warm**.

The core reference is a warm monochrome *display surface*: Fallout Pip-Boy, e-ink reader, amber-phosphor CRT, early-80s instrument readout. It is NOT paper, parchment, or manila. The cream isn't cardstock and the dark brown isn't ink — they're the lit and unlit states of a single-phosphor screen. The grain on top is a display artifact (subpixel shimmer, phosphor wear), not paper texture.

- *Serialized* — things feel like they have a part number. Certificates, manifests, badges, project IDs are rendered as readouts, not rubber-stamps.
- *Instrumented* — the interface has dials, readouts, corner brackets, barcode strips. A console you read, not a feed you scroll.
- *Warm* — the monochrome is warm, not cold. Cream-on-brown in light mode, orange-on-dark-brown in dark mode. Never cyan, never blue-on-black, never chrome.

The "stasis" name is taken literally: cryo-pod manifests, expedition gear, analog instrument panels, mission patches — but displayed on a single-color retro screen, not printed on paper.

### Aesthetic Direction

**Theme split:**
- **Light mode** is the default for the public site and dashboard — cream readout (`cream-50/100`) as the lit state, toasted brown (`brown-900`) as foreground, orange (`orange-500`) as the live accent. Think warm e-ink.
- **Dark mode** is a deliberate *room* used on the **starter projects page** (cryo-capsule catalog) and admin pages (back-of-house). It is a single-phosphor display: orange and warm-brown on dark-brown, like a Pip-Boy or amber CRT — never cyan-on-black, never pitch black.

**Established design vocabulary** (already shipping, keep using these):
- **Palette:** cream `#F5F3EF`/`#EBE8E0`/`#D5CCB7`, brown `#3B3026`/`#403529`, orange `#D95D39`/`#E86A3A`/`#FC8A58`, yellow `#E1AB55`, taupe `#9C8F88`. Orange is the ONLY saturated accent — use it sparingly, let cream/brown carry the weight.
- **Fonts:** Space Grotesk (body/UI), Departure Mono (localFont — distinctive display for readouts, labels, serials), Libre Barcode 128 & 39 (actual barcode strips as decoration — a core brand motif). Departure Mono is the project's signature display face; it's the thing that makes the site feel like Stasis and not a generic SaaS.
- **Recurring motifs:** `MagneticCorners` (corner brackets that track a target — every interactive canvas has these), `NoiseOverlay` / `PlatformNoiseOverlay` (display-surface grain — subpixel shimmer / phosphor wear, not paper texture; surfaces are never perfectly clean), `DottedLine` (dashed/dotted rules — schematic hints, ruled readouts), `HoverScramble` (text scrambles on hover — retro-terminal decoding effect), `ASCIIArt` (monospace art as first-class content), barcode font strips as serialized decoration, status pills that read like inspection readouts.
- **Selection color:** orange-on-cream. The focus ring is orange. Never replace these — they're the visual signature.

**Anti-references — explicitly NOT:**
- **Generic SaaS / Linear / Vercel dashboard.** No purple-to-blue gradients. No glass cards. No rounded-everything hovering on plain white. No icon-above-heading card grids.
- **Gradient text** (absolute ban — it's the #1 AI tell).
- **Side-stripe `border-left: 4px solid accent`** (absolute ban — dashboard slop).
- **Generic dark-mode neon** (cyan-on-#0a0a0a). Dark mode here should still feel analog — warm-tinted neutrals, not pitch black.

### Hard Rules

- **No rounded corners.** Sharp edges are the default everywhere — buttons, cards, inputs, images, modals. Circles are occasionally allowed (avatars, true dots/pills where geometry demands it) but are easy to get wrong, so default to square and only reach for a circle when it's unambiguously the right shape. Never `rounded-md` / `rounded-lg` as a reflex.
- **Departure Mono (the pixel font) needs size discipline.** Because it's a pixel font, mixing sizes produces visible "mixels" — pixel grids at different scales that don't align and look broken. Pick as few sizes as the UX allows (ideally one or two across a whole surface) and create hierarchy through other means: weight-where-available, letter-spacing, CAPS vs lowercase, color/value contrast, spacing, rules/dividers, indentation, inversion (light-on-dark blocks). If you find yourself adding a third Departure Mono size, first try to solve the hierarchy problem without one.

### Design Principles

1. **Everything has a part number.** Lean into serialization. Projects, badges, orders, certificates — give them IDs, timestamps, stamps, barcode strips. Make the user feel like they're interacting with *issued* artifacts.
2. **Console, not glass, not paper.** Surfaces have display grain (noise overlay) — a phosphor/e-ink artifact, not paper texture. Borders are hairline or dashed, never soft-shadowed. Never `backdrop-blur` as decoration. No frosted glass, no drop shadows. When in doubt, imagine the element rendered on a single-color display.
3. **Cream carries, orange accents, brown grounds.** Follow 60-30-10 strictly: cream surfaces, brown text/borders, orange reserved for live state (CTAs, selection, in-progress). When orange appears it should *mean* something.
4. **Instrumented corners, not full frames.** Use `MagneticCorners` and bracket motifs to frame interactive zones instead of boxing everything in cards. Flatten the card nesting; use composition and rhythm instead.
5. **Retro as craft, not costume.** Barcode strips, mono readouts, scramble effects, ASCII art — these are allowed because they're functional-looking, not kitsch. No CRT scanlines, no Win95 chrome, no fake-CRT glow. If a retro element doesn't pull its weight, cut it.
6. **Welcome the beginner in the copy, respect the expert in the structure.** Microcopy should be warm, specific, encouraging ("you'll earn your ticket", not "submit entry"). But the information architecture — tiers, bits ledger, badge criteria, build logs — should be legible and precise enough that an experienced maker trusts it.

### Scope Note

The landing page (`app/page.tsx`) and starter projects page (`app/starter-projects/page.tsx`) are the canonical reference surfaces — they have the full attention to detail (magnetic corners, noise, scramble, ASCII, barcode strips, orchestrated motion). Most other user-facing pages (dashboard, project detail, certificate, etc.) share the same design language but haven't reached that bar yet. When designing new features or upgrading existing ones, pull from those two pages as the reference, not from whichever page happens to be nearby.

### Admin Design Context

Admin pages share the warm-monochrome palette but are a separate **operator console** dialect — dense, keyboard-driven, no theatricality. They are dark-mode by default (the `AdminLayout` wraps everything in `bg-brown-900 font-mono`, but individual pages opt back into `font-sans` for the content). The reference surface for any new admin work is **`/admin/attendance`** (`app/admin/attendance/`). The older admin pages (`/admin/projects`, `/admin/users`, parts of `/admin/review`) predate the dialect and use Tailwind-default pastel pills (`bg-yellow-100 text-yellow-800`, `bg-gray-200`) — do not copy from those; upgrade them toward the attendance style when you touch them.

**Do not use Departure Mono on admin pages unless explicitly requested.** The page root opts back into `font-sans` (Space Grotesk) — the pixel font's size-discipline cost isn't worth it for back-of-house UI, and it makes admin tools feel theatrical when they should feel utilitarian.

**Palette (admin):** four tokens carry everything.
- Surfaces: `bg-brown-900` (page), `bg-brown-800` (cards/inputs/buttons), `bg-brown-950` (footer/statusline), `bg-brown-700/40` (hover on dark surfaces).
- Text: `text-cream-50` (headings/primary), `text-cream-100` (body), `text-cream-200` (labels/secondary), `text-cream-300` (tertiary/meta), `text-cream-400` (disabled/placeholder).
- Accent: `text-orange-500` (the tab + heading orange) and `text-orange-400` / `bg-orange-500/15` / `bg-orange-500/20` (selected & CTA — always tinted, rarely solid).
- Semantic text: `text-green-400` (success), `text-red-400` (error), `text-yellow-400` (warn), `text-pink-300` (girl-target stats). Use as **text color**, not as a pill background — semantic colors stay quiet.

**Surface rules.**
- Sharp corners everywhere (no `rounded-*`, even on inputs and pills).
- Frames are inset hairlines, never borders: `outline outline-1 -outline-offset-1 outline-cream-200/15`. Dividers are `divide-cream-200/10` or `border-cream-200/10`.
- Inset shadows and soft drop-shadows are out; one allowed shadow is the menu lift `shadow-[0_8px_24px_rgba(0,0,0,0.5)]`.
- Side accent rails on chips/cards must be a **sibling flex element** (`<span className="block w-1.5 shrink-0 bg-{tone}" />`), never `border-left` — this composes the stripe so it doesn't trip the `border-left > 1px` ban.

**Type & control tokens.**
- Buttons / chips / segmented controls: `text-xs uppercase tracking-widest font-medium px-3 py-2`, surface `bg-brown-800`, text `text-cream-200 hover:text-cream-50`, press feedback `active:scale-[0.97]`, transition `transition-[color,background-color] duration-150`.
- CTA variant: same shape, swap to `bg-orange-500/15 text-orange-400 hover:bg-orange-500/25`. Never solid `bg-orange-500` for buttons — tinted only.
- Inputs: `bg-brown-800 text-cream-50 text-xs font-medium px-2.5 py-2 outline-none focus:ring-2 focus:ring-orange-500/60 focus:ring-inset placeholder:text-cream-400`.
- Section/page heading: `text-cream-50 text-xl font-medium` (not uppercase — heading is title-cased, only labels and buttons CAPS).
- Meta/labels (status, timestamps, kbds, kanban column titles): `text-xs uppercase tracking-widest text-cream-{200,300} font-medium`.
- Tabular-nums on every count, timestamp, kbd, dollar value, ratio (`tabular-nums` baked in).

**Tables.**
- `border-separate border-spacing-0` with sticky `thead` at `bg-brown-900`; the `border-spacing-0` plus `bg-brown-800` body produces a faint grid without explicit borders.
- Zebra: alternating rows tinted `bg-brown-900/30`.
- Hover/selection: rows carry `outline outline-2 -outline-offset-2 outline-transparent`; hover is `hover:bg-orange-500/5`, selected is `bg-orange-500/15 outline-orange-500/60`. The outline collapses the row in place rather than shifting layout.

**Status dots & micro-indicators.**
- Square dots: `w-1.5 h-1.5 bg-{tone}` (NOT `rounded-full`). Use for sync state, freshness, etc.
- Touch-health dot (fresh/stale/cold) is the rare exception that uses `rounded-full` because it lives inline with body text — keep that exception narrow.
- Pulsing affordance: a separate `.attendance-dot-fresh` CSS class drives a 2.4s ease-in-out opacity pulse, gated by `prefers-reduced-motion`.

**Motion.**
- Row/card entrance is staggered via a `--row-i` CSS variable + `calc(var(--row-i, 0) * 8ms)` delay applied through `.attendance-row` / `.attendance-card`. Cap the stagger (`Math.min(idx, 14)`) so long lists don't drag.
- Easing: `cubic-bezier(0.22, 1, 0.36, 1)` (ease-out-quart). Durations 140–220ms.
- Press feedback: `active:scale-[0.97]` on every clickable chip/button.
- All custom animations sit behind a `@media (prefers-reduced-motion: reduce) { animation: none !important; }` block.

**Density & operator affordances.**
- The page is a fixed-height app surface, not a scrolling document — `attendance-page-root` does the `100dvh - chrome` math so the content fills viewport and the bottom statusline sits flush.
- Bottom statusline shows `showing N of M` on the left and inline `<Kbd>` chips on the right (`bg-brown-800 text-cream-200 text-xs tabular-nums px-1.5 py-px`).
- Keyboard nav is first-class: `/` focuses search, `j`/`k` or arrows traverse rows, `Enter` opens, `n` creates, `v` cycles views, `?` opens help, `Esc` peels back state (modal → adding → detail → highlight → search). Build new admin tools to the same vocabulary.
- Filters mirror to the URL (debounced `router.replace`), so deep links survive reload and middle-click on tabs/segments opens that view in a new tab (anchors with intercepted left-click).
- Admin chrome (header + tab bar in `AdminLayout`) carries `data-admin-chrome`; pages that want full-bleed dense layouts can set `.hide-admin-chrome` on `<html>` to collapse it (toggled per-page, persisted in localStorage).

**Composition patterns to reach for.**
- `SegmentedView`: a single `bg-brown-800` bar with `outline-cream-200/10` and `border-l border-cream-200/10` between segments (no fake spacer pixels). Active segment uses the orange-tint treatment.
- `SyncMenu`-style consolidated dropdowns: one trigger that summarises N background syncs, opens a `divide-cream-200/10` list, each row has its own run button + last-run relative time + inline result line + dismissable error.
- "Funnel chip" / readout: dark surface, left accent rail, label on top (CAPS micro-label), big tabular-nums value below, optional secondary stat in a muted color.
- Hover-reveal alternate readout (see `GirlTargetChip`): two stacked panels translated under a `group-hover` to swap one stat for another in place — preferred over tooltips for primary stats.

**Anti-patterns specific to admin** (in addition to the global bans):
- No solid `bg-orange-500` button fills. Tinted only.
- No `rounded-*` on admin controls.
- No pastel Tailwind-default status pills (`bg-yellow-100 text-yellow-800` etc.) — that's the legacy admin look being phased out.
- No icon-above-heading card grids, no stat hero blocks with gradient accents, no glassmorphism.
- Don't introduce new font weights/sizes when hierarchy can come from CAPS + tracking + cream-shade contrast.
